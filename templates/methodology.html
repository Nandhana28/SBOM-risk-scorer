Methodology & audit reference

The complete risk policy on one page. Every threshold below is read live from sbom/config.py — this documentation cannot drift from the running engine.

Design principle

Risk decisions are made by deterministic rules, not a trained model. For a governance tool this is deliberate: an auditor can reproduce and challenge every finding from the evidence, the same SBOM always yields the same result, and there is no over-fitting to a particular sample. Thresholds are chosen on security-domain principle (industry CVSS bands, a 2-year staleness convention), so the engine generalises to SBOMs it has never seen.

1 · License compatibility

A dependency is flagged when its declared licence is marked compatible_with_proprietary = false in the licence matrix and the application ships under a proprietary model — or when it declares no licence at all (UNKNOWN). This rule is exact against the ground truth (100% recall, 0 false positives).

2 · Vulnerability matching (version-aware)

A CVE affects a dependency when the installed version is below the advisory's fixed_version, or when no fix exists yet. This is the same package+version test used by Dependabot, pip-audit and Grype. (The dataset's affected_versions field is unreliable and is deliberately not used.)

3 · Maintenance

A dependency is flagged UNMAINTAINED when its last release predates the reference date ({{ cfg.REFERENCE_DATE }}) by more than {{ cfg.MAINTENANCE_YEARS }} years.

5 · Scoring (transparent & additive)

Each dependency's score follows Option C's prescribed shape, with every point attributable to a named component:

score = Σ(matched-CVE CVSS, ×{{ cfg.UNPATCHED_MULTIPLIER }} if unpatched) + license_penalty + maintenance_penalty, then × depth-weight (direct {{ cfg.DEPTH_WEIGHT.direct }}, transitive {{ cfg.DEPTH_WEIGHT.transitive }})

license_penalty by licence risk level: CRITICAL {{ cfg.LICENSE_PENALTY.CRITICAL }} · HIGH {{ cfg.LICENSE_PENALTY.HIGH }} · MEDIUM {{ cfg.LICENSE_PENALTY.MEDIUM }} · unknown {{ cfg.LICENSE_UNKNOWN_PENALTY }}. maintenance_penalty {{ cfg.MAINTENANCE_PENALTY }}. Per-application score = Σ(dependency scores) × criticality weight (CRITICAL {{ cfg.CRITICALITY_WEIGHT.CRITICAL }} · HIGH {{ cfg.CRITICALITY_WEIGHT.HIGH }} · MEDIUM {{ cfg.CRITICALITY_WEIGHT.MEDIUM }} · LOW {{ cfg.CRITICALITY_WEIGHT.LOW }}).

6 · Architecture

SBOM data lives in SQLite (each upload isolated under its own scan id, with a durable audit log of every run). The engine reads from the database, applies the rules above, persists findings, and the Flask layer renders views. See Validation for measured accuracy against the labelled ground truth.
